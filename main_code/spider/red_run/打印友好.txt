
  一、终端上“批次面板”的展示方式

  - 整体是“按批次”的展示：
      - 每一批都有一个清晰的标题行，能看出来“现在是在
        第几批 / 总共几批”。
      - 下面是这一批所有学号的列表，每个学号一块信息，
        内容包括：学号、密码（或脱敏后的密码）、当前耗
        时、当前状态、进度百分比等。
  - 这一批的学号列表在开始时就“定死”：
      - 一旦这一批开始执行，这一批里有哪些学号、顺序是
        怎样的，就不要再改变；
      - 中途不要突然多出新行、也不要改变顺序，视觉上一
        个学号对应固定的位置。
  - 执行过程中，这个“批次面板”要占据终端的一块固定
    区域：
      - 在这一块区域里，数字和文字（时间、进度、状态）
        可以不断刷新、变化；
      - 但这块区域本身不要随着时间不断往下滚动增加新
        行，而是“原地更新”。
  - 当前批次还在跑的时候：
      - 只更新每个账号那几行里的“变动字段”：耗时、进
        度、状态等；
      - 标题上显示“第几批 / 总批次”的那一行应该是稳定
        的，不要每秒再额外打印一遍导致向下滚。
  - 等这一批所有账号都跑完了：
      - 把这一批的面板状态“定格”在屏幕上；
      - 然后光标移动到下面空出一些位置，再开始显示下一
        批的面板。
      - 从你的视角看，就是：一屏一屏地往下“翻批次”，而
        不是一行一行地不断往下刷。

  二、终端输出 vs 日志文件：角色分工

  - 终端输出的作用：
      - 更像一个“实时监控大屏”，让你一眼就能看到当前每
        个账号在干什么，跑到哪一步了。
      - 重点是“当前状态”和“进度感”，不关心太细的过程
        细节。
  - 日志文件的作用：
      - 更偏向“事后审计”和“排错记录”，方便回看某个批次
        整体跑得怎么样、哪些账号成功、哪些账号失败、为
        什么失败。
      - 日志里不应该被一堆每秒钟的过程信息淹没。
  - 所以你希望做到：
      - 终端面板可以比较频繁地刷新（每秒、几秒一次），
        但这些刷新不需要逐条记录到日志里。
      - 日志文件里尽量只保留关键节点的信息，比如：
          - 某个账号最终是成功、密码错误、已经跑过、网
            络异常、结束失败等最终状态；
          - 某一批开始处理时的简短说明；
          - 某一批结束后的完整总结快照；
          - 以及真正的异常 / 错误信息。
      - 像倒计时时每隔几秒打印一次“还剩 xx 分钟”这种过
        程性文字，你希望只在终端看到，不要写进日志。

  三、批次节奏与日志节奏

  - 批次的节奏：
      - 先从数据源中拿到一批账号；
      - 在终端上固定出一块面板区域，列出这一批的所有
        账号；
      - 一边跑一边更新这一块区域里的耗时、进度、状态；
      - 整个批次跑完后，面板显示成“已完成”状态，然后在
        终端向下挪出位置，开始下一批。
  - 日志的节奏：
      - 一批开始时简单记一笔“第几批 / 有多少账号”；
      - 过程中只在出现异常或比较关键的节点时记必要的信
        息，不要每次小变动都记；
      - 一批结束时，把当前这一批的整体情况（就类似于你
        在终端看到的那块面板的最终状态）作为一个快照写
        入日志。
      - 最好做到：一个批次对应日志里一段清楚的“摘要”，
        而不是散落着一大堆过程碎片。

  四、错误账号、已完成账号的处理

  - 对已经确认密码错误的账号：
      - 再跑的时候要自动跳过；
      - 终端上可以简单提示“跳过了多少个已知密码错误
        账号”；
      - 日志里可以有一条概述，比如“根据某个记录文件跳
        过了 N 个密码错误账号”，不需要对每一次跳过做大
        量过程描述。
  - 对当天已经跑完挑战的账号：
      - 在这一批的面板里，要能看出来这是“已完成”的状
        态，而不是看起来像失败或异常；
      - 日志里也可以把这类账号的最终状态记录出来，用于
        之后审查。

  五、整体体验目标

  - 终端体验：
      - 像一个“分批次的进度看板”：
          - 一屏一屏按批次显示；
          - 每批里的账号状态在固定排版下不断刷新；
          - 不再出现之前那种疯狂往下滚、第一二个账号重
            复打印、批次标题行一秒一刷的怪异现象。
  - 日志体验：
      - 像一份“干净的运行记录”：
          - 看日志能快速了解每一批的开始、结束时间、大
            致情况以及每个账号的最终结果；
          - 真正遇到错误时能看到有用的错误信息；
          - 不需要在成千上万条过程性信息中费劲翻找。

  如果按这份描述来对准：你核心追求的是“终端上稳定清晰
  的批次面板 + 日志里简洁有用的结果记录”，而不是“看着
  很热闹但信息噪音很大的刷屏输出”。
